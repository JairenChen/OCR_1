# Learning Notes

## 什么是抽帧

视频本质上是一连串图片。抽帧就是按一定间隔从视频里取出单张图片，例如每 1 秒取一帧。这样我们可以先把“视频 OCR”简化成“图片 OCR”。

## FPS 和采样间隔

FPS 表示视频每秒有多少帧。30 FPS 表示 1 秒有 30 张画面。

本项目的 `sampling.interval_seconds` 是采样间隔。比如设置为 `1.0`，就是每秒抽一帧；设置为 `0.5`，就是每半秒抽一帧。

间隔越小，漏字概率越低，但 OCR 次数越多，速度越慢。

## OCR 检测和 OCR 识别的区别

OCR 通常分两步：

1. 文本检测：找出图片里哪里有文字，输出文字框。
2. 文本识别：把文字框里的图像内容识别成字符串。

PP-OCRv6_medium 里我们显式使用：

- `PP-OCRv6_medium_det` 做文本检测
- `PP-OCRv6_medium_rec` 做文本识别

## 为什么视频字幕需要去重

同一句字幕通常会在视频里停留 1 到 3 秒。如果我们每秒抽一帧，同一句话会被识别多次。

去重与合并的目标是把这些连续重复结果合成一个时间段：

```json
{
  "start_time": "00:00:01.000",
  "end_time": "00:00:03.000",
  "text": "同一句字幕"
}
```

## 可选文字区域过滤

画面里可能有 Logo、角标、商品文案、按钮文字。字幕通常出现在画面下方，所以可以只保留画面底部区域的文字。

当前配置默认不启用区域过滤。需要时可以在 `configs/default.yaml` 中设置：

```yaml
filter:
  text_region:
    enabled: true
    y_min: 0.45
    y_max: 1.0
```

这里的坐标是比例，不是像素。`y_min: 0.45` 表示从画面高度 45% 的位置开始，到底部结束。

## 文字区域过滤的判断流程

PP-OCRv6 会先检测画面中所有可能的文字框。区域过滤不是改变模型，而是在模型输出之后做规则判断：

1. 清洗文本：去掉空字符串和多余空格。
2. 置信度过滤：低于 `filter.min_confidence` 的结果先丢掉。
3. 区域过滤：计算文字框中心点，只保留中心点落在 `text_region` 内的文字。
4. 合并字幕：把过滤后的连续重复文本合并成时间段。

第一版采用“中心点是否在区域内”，因为它简单、直观、容易调参。后续如果遇到很长的花字或斜着的文字框，可以升级成“文字框和文字区域的重叠比例”。

## Debug 图片怎么看

debug 图片用于回答三个问题：

- OCR 是否找到了正确文字框。
- 文字区域是否设置得太高或太低。
- 置信度阈值是否过滤掉了有用字幕。

推荐颜色约定：

- 绿色框：通过过滤，会进入后续字幕合并。
- 红色框：被过滤，不会进入字幕合并。
- 蓝色框：当前配置的字幕候选区域。

如果红色框里有真正文字，说明过滤规则过严，通常要降低 `min_confidence` 或调整 `text_region`。如果绿色框里有不想要的水印、角标或 UI 文案，说明过滤规则过宽，通常要收窄文字区域。

## 当前目标：整页文字提取

我们当前不再默认只提取字幕，而是提取整个画面里的文字。

文字区域过滤仍然有用，但它变成了一个可选工具。默认配置应保持：

```yaml
filter:
  text_region:
    enabled: false
```

这样 PP-OCRv6 检测到的标题、商品文案、字幕、角标和 UI 文字都会先被保留下来，再根据置信度和后续规则处理。

## 什么是版面阅读顺序

整页 OCR 的一个常见问题是：模型找到文字框的顺序，不一定是人眼阅读顺序。

人眼通常按：

```text
从上到下
同一行从左到右
```

阅读。因此我们加入了版面排序：

1. 根据文字框的 y 坐标判断它属于哪一行。
2. 同一行内根据 x 坐标排序。
3. 最后把每一行拼成当前帧的文本。

这一步不会改变 OCR 识别结果，只改变输出顺序。

## 整页文字为什么也需要跨帧合并

整页文字通常会在视频里停留一段时间。例如商品页、教程页面、PPT 页面、短视频贴片文案，都可能连续多帧基本不变。

如果每一帧都输出一次，最终 TXT 会充满重复内容。跨帧合并会把连续相似的页面快照合成一个时间段：

```json
{
  "start_time": "00:00:02.000",
  "end_time": "00:00:06.000",
  "text": "当前页面上的主要文字"
}
```

这和字幕合并的原理类似，只是合并对象从“一句字幕”变成了“一页文字快照”。

## 什么是噪声文本过滤

OCR 会尽力识别所有看起来像文字的区域，但有些结果对我们没有意义，例如：

- 单独的符号：`*`
- 重复标点：`---`
- 置信度很低的乱码
- 空字符串

噪声过滤就是在 OCR 之后，把这些明显无用的结果去掉。

整页文字提取里，噪声过滤必须保守。因为很多短文本其实很重要，例如 `50%`、`OFF`、`MIN`、价格、倒计时。当前默认不会过滤纯数字文本，避免误删这些信息。

配置示例：

```yaml
filter:
  noise:
    enabled: true
    min_text_length: 2
    drop_symbol_only: true
    drop_repeated_punctuation: true
    drop_numeric_only: false
```

如果后续只想保留自然语言句子，而不想要倒计时或价格，可以再考虑打开 `drop_numeric_only`。

## 什么是位置感知跨帧合并

跨帧合并的目标是把连续多帧中重复出现的文字合成一个时间段。

只看文本会有风险。例如同一个词 `LIMIT` 可能先出现在顶部，后出现在底部。它们文本一样，但不一定是同一个画面文字对象。

位置感知合并会同时看三件事：

```text
文本是否相似
时间是否连续
位置是否接近
```

位置接近主要看：

1. 两个文字框是否有足够重叠，也就是 IoU。
2. 两个文字框中心点距离是否足够近。

IoU 可以理解为两个框重叠面积占合并面积的比例。值越大，说明两个框越重合。

中心点距离适合处理 OCR 框大小有轻微变化的情况。即使两个框没有完全重叠，只要中心点很接近，也可以认为是同一个位置。

这一步不是复杂目标跟踪，但已经能避免很多“同样文本移动到不同位置还被合并”的问题。

## 什么是轻量状态机

旧的 `snapshot` 合并策略把一整帧过滤后的文字当作一个页面快照来比较。它适合“整页文字整体变化”的场景，但当同一画面里有标题、价格、角标、字幕等多个文字块时，一个文字块变化就可能影响整帧快照。

轻量状态机 `line_state_machine` 把每个 OCR 文本框看成一个正在被跟踪的文字对象：

```text
active：文字对象当前还在画面中
lost：这一帧没匹配到，但还没有超过允许间隔
closed：超过允许间隔仍没出现，输出为一个时间段
```

匹配一个文字对象时，状态机会同时看：

```text
文本是否相似
位置是否接近
距离上次出现是否没有超过 max_gap_seconds
```

这样可以处理短暂漏检。例如某个词在 `1.0s` 出现，`1.5s` 被 OCR 漏掉，`2.0s` 又出现，只要没有超过 `merge.max_gap_seconds`，它仍会被合并成同一个时间段。

默认配置：

```yaml
merge:
  strategy: line_state_machine
  max_gap_seconds: 1.5
  use_position: true
```

如果想回到旧的整页快照式输出，可以改成：

```yaml
merge:
  strategy: snapshot
```

需要注意：状态机默认按“文本框/文本行”输出片段，所以 SRT 会更细；`snapshot` 默认按“整帧页面快照”输出片段，所以 SRT 更像页面级摘要。

## 为什么需要运行统计报告

视频 OCR 的效果不仅取决于模型，也取决于抽帧间隔、视频分辨率、GPU/CPU 环境、过滤阈值和合并参数。

如果每次只看最终 SRT，很难判断问题来自哪里。例如：

- 抽帧数量太少，可能漏掉短暂出现的文字。
- OCR 耗时过长，可能需要调大抽帧间隔或优化采样策略。
- 文字框数量异常少，可能是 OCR 阈值过高或模型没有正确加载。
- 片段数量异常多，可能是跨帧合并条件过严。
- 环境版本变化，可能导致同一视频跑出不同结果。

因此项目会在每次运行后导出：

```text
*_report.json
*_report.txt
```

`*_report.json` 更适合程序读取和后续分析；`*_report.txt` 更适合人工快速查看。

报告里重点关注这些字段：

```text
视频信息：分辨率、FPS、总帧数、时长
抽帧信息：采样间隔、实际抽帧数
OCR 统计：处理帧数、有文字帧数、文字框总数、平均置信度
合并统计：合并后片段数、覆盖时长、平均片段时长
用时统计：抽帧耗时、OCR 耗时、合并耗时、总用时
环境信息：Python、Conda、PaddleOCR、PaddlePaddle、Torch、OpenCV、NumPy
```

这份报告的作用不是替代 debug 图片，而是帮助我们快速判断一次实验是否“正常”。debug 图片负责看视觉细节，统计报告负责看整体健康度。

## 什么是自适应边界细化

固定间隔抽帧的缺点是时间边界不够准。例如每 1 秒抽一帧，文字真实在 `1.30s` 出现，但第一次抽到它可能是 `2.00s`，最终 SRT 就会偏晚。

自适应边界细化的思路是：

```text
先粗采样，找到大概有哪些文字片段
再只在每个片段的开始和结束附近密集采样
用密集采样结果修正 start_time 和 end_time
```

当前实现采用简单线性扫描：

```text
粗采样片段：2.00s -> 5.00s
细化窗口：0.50s
细化间隔：0.20s

开始边界搜索：1.50s, 1.70s, 1.90s, 2.00s
结束边界搜索：4.50s, 4.70s, 4.90s, 5.00s
```

如果在 `1.70s` 已经能匹配到同一段文字，就把开始时间从 `2.00s` 修正为 `1.70s`。如果在 `4.80s` 后不再匹配，就把结束时间从 `5.00s` 修正到更接近真实消失的时间。

判断“是否是同一段文字”时，项目会复用已有规则：

```text
文本相似度
位置相似度
置信度和噪声过滤
```

这不是实时视频系统，也不是复杂目标跟踪。它更像离线处理中的“二次校准”：先用较低成本找到大概范围，再对边界花更多 OCR 成本换取更准的 SRT 时间。

配置示例：

```yaml
sampling:
  strategy: adaptive_boundary
  coarse_interval_seconds: 0.5
  refine_window_seconds: 0.5
  refine_interval_seconds: 0.2
  refine_ocr_batch_size: 1
  refine_max_extra_ocr_frames: 80
  boundary_text_similarity_threshold: 0.85
```

如果想对比旧算法，可以改回：

```yaml
sampling:
  strategy: fixed
```

自适应边界细化会增加额外 OCR 次数，所以运行会变慢。每次运行后的 `*_report.txt` 会记录额外抽帧数、额外 OCR 帧数、额外 OCR 预算和边界细化耗时，方便判断这次优化是否值得。

性能模式下，项目默认关闭 debug 图片，并关闭文本行方向分类：

```yaml
sampling:
  decode_mode: seek
  save_frame_images: false
  keep_frame_images_in_memory: true

ocr:
  batch_size: 8
  use_textline_orientation: false

ocr_cache:
  enabled: false
  image_diff_threshold: 0.5
  max_pixel_diff_threshold: 12.0
  image_size: 96

debug:
  enabled: false
```

`save_frame_images: false` 表示正式 OCR 时不再把每一帧先写成 jpg，再让 OCR 从磁盘读回来；而是把 OpenCV 解码得到的 numpy 图像直接传给 PaddleOCR。这样不改变 OCR 模型，也不降低识别准确度，只是减少磁盘 I/O。

`ocr.batch_size: 8` 表示一次把多帧图像组成 list 传给 PaddleOCR。它不会改变识别结果的含义，只是让 GPU 一次处理更多输入，减少一帧一帧调用模型的调度开销。显存较小时可以调小，例如 `4` 或 `2`；如果要观察相似帧 OCR 复用缓存，可以临时设成 `1`。

`sampling.refine_ocr_batch_size: 1` 是边界细化阶段的额外 OCR 批大小。边界细化不能简单把所有候选帧一次性 OCR，因为它需要边扫描边判断“文字是否已经出现/消失”。当前默认保持逐帧按需细化；如果以后要做性能实验，可以临时调成 `2` 或 `4`，再观察报告里的 `extra_ocr_frames`、`skipped_by_budget` 和耗时是否真的变好。

`decode_mode: seek` 表示按目标时间点随机定位抽帧。当前项目默认每 0.5 秒抽一帧，属于稀疏采样，实测 `seek` 更稳。`sequential` 和 `auto` 仍然保留，适合以后采样更密时做对比。

## 什么是批量处理模式

批量处理不是新的 OCR 算法，而是把已经跑通的单视频 pipeline 自动应用到一个文件夹里的多个视频。

当前批量模式做三件事：

```text
扫描目标文件夹里的视频文件
为每个视频建立独立输出子目录
跑完后把生成的 SRT 复制回原视频所在目录
```

示例：

```powershell
python -m src.main --video-dir E:\OCR\test_vedio --output-dir E:\OCR\data\batch_outputs
```

输出结构大致是：

```text
batch_outputs/
  1/
    1.srt
    1_segments.json
    1_report.txt
    frames/
  another_video/
    another_video.srt
    another_video_segments.json
    another_video_report.txt
    frames/
  batch_summary.json
  batch_summary.txt
```

批量模式会复用同一个 PaddleOCR 引擎实例，避免每个视频都重新加载 PP-OCRv6 模型。单个视频失败时，批量流程会继续处理后面的视频，并把错误写入 `batch_summary.json` 和 `batch_summary.txt`。

## 什么是相似帧 OCR 复用

视频里经常会出现连续多帧画面几乎不变的情况。如果每一帧都完整调用一次 PP-OCRv6，就会重复做很多相同工作。

相似帧 OCR 复用的思路是：

```text
先把当前帧缩小成一张很小的灰度图
和上一帧的小灰度图比较差异
如果平均差异和最大像素差异都很小
就认为画面基本没变，直接复用上一帧的 OCR 结果
```

这里复用的是“已经过滤和排序后的 OCR 结果”，不是修改模型，也不是降低 OCR 精度。它只是减少重复调用 PaddleOCR 的次数。

为了降低漏检风险，当前实现比较保守：除了看平均差异，还会看最大像素差异。这样可以避免一个很小的新文字被全画面平均值稀释掉。debug 画框模式下会自动关闭该缓存，因为 debug 的目的就是观察每一帧真实的 OCR 检测框。

当前默认配置下优先使用批量 OCR，相似帧缓存默认关闭。原因是很多短视频画面一直在动，相似帧很少，而批量 OCR 对这类视频更稳定。如果视频是 PPT、录屏、静态页面，可以把 `ocr.batch_size` 改成 `1`、把 `ocr_cache.enabled` 改成 `true`，再观察报告里的 `ocr_cache.reused_frames` 是否明显增加。

如果要排查 OCR 框位置、保留抽帧图片或处理旋转文字，可以临时打开它们；正式批量处理时建议保持关闭。
